I have selected the Automated Kanban template. It aligns with my project needs because it automatically moves issues across the board as

## 3.Kanban Board Explanation

What is a Kanban Board?

A Kanban board is a visual management tool that helps a person or team see exactly what needs to be done, what is currently being worked on, and what is finished. It turns a "to-do list" into a visual flow.

How My Board Works:
Visualizes Workflow: Each column (To Do, In Progress, Testing, Done) represents a stage in the software development life cycle. I can see at a glance where every user story is.

Limits Work-in-Progress (WIP): By limiting the number of tasks in the "In Progress" column (e.g., maximum of 2 tasks), I avoid taking on too much at once, which prevents burnout and ensures I actually finish one thing before starting another.

Supports Agile Principles: It supports Continuous Delivery because as soon as a task hits "Done," it's ready for use. It also allows for Adaptability, as I can easily move a task back to "To Do" if requirements change.

## Reflection: Customizing the Workflow

The biggest challenge I faced was the Paradox of Choice. GitHub offers several templates, and at first, I wasn't sure if I should use the "Basic Kanban" or the "Team Planning" one. I initially chose the Basic version, but I quickly realized that without automation, I would have to spend too much time manually dragging cards around every time I made a small change.

Customizing the columns also felt like a balancing act. I felt some internal resistance when adding the "Testing" and "Blocked" columns. Part of me felt like I was making the board too complicated for a one-person project. However, I realized that without a "Blocked" column, I would just leave a task in "In Progress" for days, which made it look like I was working when I was actually just stuck. Adopting these extra columns forced me to be honest about my progress and where I was procrastinating.

Comparison: GitHub Projects vs. Jira vs. Trello

In my experience, Trello is the friendly version of a Kanban board. It is great for a simple shopping list or a basic project, but it lacks the deep technical connection to the actual code.

Jira, on the other hand, felt like overkill for this assignment. It has so many buttons and settings that I spent more time configuring the tool than actually planning my tasks.

GitHub Projects is just right. Because my code and my User Stories are already on GitHub, I don't have to jump between different websites. Being able to link an Issue directly to a card on the board makes traceability much easier than it would be in Trello. For an IT student, having the code and the management board in the same ecosystem is a huge time-saver.
